# Kenos Assistant · 差距与优化空间分析

**日期：** 2026-07-21  
**依据：**  
- 30 项单轮功能 QA（`kenos-ai-chat-llm-qa-2026-07-21`）  
- 12 场景自适应多轮 QA + **Grok 4.5 独立重评**（`kenos-ai-chat-multiturn-deep-qa-2026-07-21`）  
- 产品实现：`apps/aios/src/lib/chat.svelte.js`、`localai.js`、`prodWriteGuard.core.js`  

**采信成绩：** 多轮 **10 pass / 2 partial / 91.7%**（Grok 4.5）；单轮常见能力 **~98%**。  
**总判断：** 「会聊天 / 能多轮办事」已成立；相对 GPT/Claude 的差距集中在 **精细指令、首轮克制、模型天花板、工具生态可靠性、生产写入与安全纵深**。

---

## 1. 差距地图（按层）

```
┌─────────────────────────────────────────────────────────┐
│ A. 对话行为层（本轮实测直接打出）                          │
│    精细指令 · 首轮过载 · 字数约束 · 改写结构 · 截断        │
├─────────────────────────────────────────────────────────┤
│ B. 产品架构层（代码可达，部分未 E2E）                       │
│    有图关工具 · 联网 Bridge · Action fail-closed          │
├─────────────────────────────────────────────────────────┤
│ C. 模型与基础设施层                                        │
│    llm-fast 35B 天花板 · 无独立 moderation · 采样截断     │
└─────────────────────────────────────────────────────────┘
```

对用户体感：A 天天碰到；B 在复杂任务/生产环境碰到；C 决定「刁钻题 / 长文深度」上限。

## 2. 对话行为层差距（Grok 4.5 重评）

### 2.1 已确认的硬差距（partial）

| ID | 现象 | 对标 GPT/Claude | 根因假设 | 优化方向 |
|----|------|-----------------|----------|----------|
| **MT01** | 「先别写代码」仍输出字段/函数名/SQLite | 会先给阶段与验收，实现细节后移或先确认 | system prompt 强调「具体、可执行」但**缺少「负向约束优先级」**；模型默认工程师模式 | Prompt：加入「约束优先级：格式/禁令 > 完整性 > 详细程度」。首轮规划模板：目标→里程碑→风险，默认不写 API/函数名 |
| **MT12** | 「更短/更具体各改一版」只给三行更短标题；首轮「提升 50%」无依据 | 会成对输出或标注维度；不编造指标 | 多约束指令解析弱；营销文案易幻觉数字 | Prompt：多约束时强制列清单自检。写作：无用户给定数据禁止百分比。后处理：检测「各改一版/分别」类结构 |

### 2.2 通过但仍低于主流体验（soft gaps）

| ID | 维度 | 现象 | 优化杠杆 |
|----|------|------|----------|
| **MT10** | relevancy 3 | 焦虑开场先甩 12 周计划+鸡汤，用户说「不要鸡汤」才收敛 | **首轮意图路由**：情绪/阻塞类 → 先给「今天一件事」；规划类才给长计划 |
| **MT04** | relevancy 4 | 模糊「健身」→ 过长问卷 | **澄清预算**：最多 3 个关键问题；或给 1 个假设方案 +「不对就改」 |
| **MT05** | relevancy/consistency | 「只要清单」仍附说明；argv vs commander 混叙 | **格式服从**：检测到「只要/仅/不要其他字」→ 提高遵从权重；方案选定后禁止并列互斥主路径 |
| **MT03** | completeness | ≤50 字偶发 53 | 字数约束可在本地做 **length check + 一句重写**（小模型二次压缩） |
| **MT01/MT07** | 完整性 | 回复在句中截断 | 产品已有 `maxTokens: 4096/8192`；QA 用了 700。产品侧关注 **finish_reason=length 时自动续写**（代码里已有续写路径，需确认 UI 提示是否够明显） |

### 2.3 已经对齐、不必优先改的

记忆召回（MT08）、角色边界（MT06）、拒答转向（MT07）、事实纠错（MT09）、代码迭代（MT02）、附件连续追问（MT11）、需求切换语言（MT05 主路径）——这些是相对 GPT/Claude **基础档可用** 的护城河，优化 ROI 低于上面 soft/hard gaps。

## 3. 产品架构层差距（相对 GPT/Claude）

| # | 差距 | 证据 | 用户影响 | 改进空间 |
|---|------|------|----------|----------|
| 1 | **有图即禁用 tools** | `chat.svelte.js`: `useTools = settings.tools && !useVision` | 截图提问无法同时查笔记/算数/搜网 | 中期：图文多模态 + tool 同轮；短期：有图时允许白名单只读工具（`calculate`/`search_notes`/`search_memory`） |
| 2 | **联网非一等公民** | 依赖 Chrome Bridge / 不稳定 `web_search` 代理 | 「此刻股价/天气」体验碎片化 | Bridge 健康检查 UX；失败时明确降级文案（已有部分）；考虑内建稳定检索路径 |
| 3 | **生产 Action 写入 fail-closed** | `prodWriteGuard`：read canary / 默认阻断 | 对话里「帮我加待办」可能叫了工具却写不进 | 产品决策：writer-canary 分阶段开；或 UI 明确「演示模式只读」 |
| 4 | **System prompt 很长** | `buildSystemPrompt` 工具/笔记/Focus/记忆大段 | 小模型易忽略尾部细则或首轮过载 | 已有「刻意精简记忆指令」经验；可对 **行为纪律**（禁令、格式、澄清预算）做短置顶块，工具手册按需注入 |
| 5 | **默认模型 llm-fast** | `DEFAULT_MODEL = 'llm-fast'`（35B MoE） | 刁钻多约束、创意深度不如 Opus/GPT-5 | 路由：检测多约束/长文/规划 → 建议或自动升 `llm-quality`/`llm-deep` |

## 4. 模型与安全层差距

| # | 差距 | 说明 | 改进空间 |
|---|------|------|----------|
| 1 | **质量天花板** | 本地 35B vs 云端前沿；MT12 数字幻觉、MT01 指令边界属典型小/中模型行为 | 难度路由升模；关键写作用 quality 档 |
| 2 | **无独立 moderation** | 安全靠基座拒答（MT07 本轮过了） | 轻量本地分类器或规则层挡高风险；审计日志 |
| 3 | **同族自判不可用** | llm-fast 判自己 100%/4.92 | 评测固定用独立 judge（本次 Grok 4.5）；回归勿信自嗨分 |
| 4 | **复读/截断风险** | `localai.js` 已有复读熔断；思考模式另有关思考重试 | 对 `finish_reason=length` 强化续写提示；监控熔断率 |

## 5. 优化优先级（建议路线）

### P0 · 两周内可做（高感知 / 低风险）

1. **System prompt 置顶「行为纪律」短块**（10–15 行）  
   - 约束优先级：禁令/格式 > 完整性 > 详细  
   - 「只要/仅输出/不要开场白」→ 严格服从  
   - 无数据不编造百分比/精确股价  
   - 模糊请求：最多 3 问或给假设方案  
   - 情绪/阻塞：先给今日一个动作，再问是否要长计划  
2. **多约束改写模板**（标题/文案类）  
   - 检测到「分别/各/更短与更具体」→ 强制表格或成对列表  
3. **finish_reason=length 体验**  
   - 确认续写入口可见；截断时 toast「继续生成」

### P1 · 产品体验跃迁

4. **意图首轮路由**（规则或小分类器）  
   - `anxiety/blocked` / `plan` / `code` / `format-strict` / `clarify`  
5. **字数硬约束后处理**（≤N 字失败则 tiny/fast 重写一刀）  
6. **有图时白名单只读工具**（不必等完整 vision+tools）  
7. **难度自动升模**提示或静默切换 quality

### P2 · 平台级（对标 ChatGPT/Claude 产品形态）

8. Vision + tools 同轮  
9. Writer-canary / 生产 Action 开放策略  
10. 独立 safety 层  
11. 稳定联网一等公民路径

| 动作 | 工时感 | 预期抬升 |

## 7. 不该做什么（避免无效优化）

- **不要**再堆单轮「会不会自我介绍」类用例当主指标——已饱和。  
- **不要**用同一本地模型自判当发布门禁。  
- **不要**为追 GPT/Claude 创意深度而默认全程 80B——延迟会毁本地助手价值；应用 **路由** 而非全局升模。  
- **不要**在未开 writer-canary 时宣传「对话即可写入 Planner」——与 fail-closed 现实冲突。

## 9. 一句话给决策

Kenos Assistant 的差距 **不是「能不能聊」**，而是：  
1）**服从精细/负向约束与首轮克制**（prompt + 轻路由即可大幅改善）；  
2）**图文工具、联网、生产写入、安全纵深**（平台债）；  
3）**本地模型天花板**（用升模路由补，不靠幻想追平 Opus）。
